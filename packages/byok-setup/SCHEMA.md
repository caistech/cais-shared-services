# @caistech/byok-setup — Manifest Schema (v0.2)

Canonical schema for `byok.config.json`. One manifest per product declares every
key an operator must supply; the wizard (CLI now, web later) renders it, validates
input, generates secrets, and distributes values to their destinations.

**Status:** spec only — package not yet built (scope C, Lane A). This schema is the
output of `/plan-eng-review` (2026-05-24) plus authoring the first two real manifests.

**Proving manifests:**
- `preflight/byok.config.json` — the original 17-service base (proved the client-facing fields).
- `mmc-application/byok.config.json` — a 30-service product that surfaced the v0.2 additions below.

---

## Top-level shape

```jsonc
{
  "product": "mmc-application",
  "version": "0.1.0",
  "description": "...",
  "destinations": {
    "env-file": { "path": ".env.local" },
    "vercel":   { "targets": ["production", "preview", "development"] }
    // "supabase": { ... }   // deferred adapter
  },
  "services": [ /* one entry per env var — see fields below */ ]
}
```

## Per-service fields

| Field | Type | Origin | Meaning |
|---|---|---|---|
| `id` | string | base | Stable slug for the entry. |
| `name` | string | base | Human label shown in the wizard. |
| `envVar` | string | base | The environment variable written. |
| `required` | boolean | base | Hard-required. **Superseded by `requiredIf` when present** (finding 2). |
| `description` | string | base | One line: what the key is for. |
| `signup` / `keyPage` | url | base | Where to create an account / find the key. |
| `instructions` | string[] | base | Step-by-step how-to-get. Pasted verbatim by the wizard. |
| `keyFormat` | regex | base | Paste-time shape check. **Necessary but NOT sufficient — see Validation.** |
| `autoGenerate` | `random-32\|48\|64` | base | Wizard generates the value. See `secretClass`. |
| `exclusiveGroup` | string | base | Members are either/or (e.g. `llm`: Anthropic OR OpenRouter). |
| `costNote` | string | base | Spend expectation, set against the operator (BYOK). |
| **`scope`** | `public\|secret\|config\|build-time` | v0.2 (review) | `public` = bundled to client (`NEXT_PUBLIC_*`); `secret` = server-only; `config` = non-secret setting. Drives logging/masking. |
| **`destination`** | string[] | v0.2 (review) | Which adapters receive it (`env-file`, `vercel`, `supabase`). Defaults to all in top-level `destinations`. |
| **`provider`** | string | v0.2 (finding 3) | Groups entries from the same account/dashboard so the operator sets it up once. |
| **`phase`** | `build\|runtime\|post-deploy` | v0.2 (finding 1) | When the key is needed/obtainable. Drives wizard pass ordering. |
| **`dependsOnDeploy`** | boolean | v0.2 (finding 1) | Cannot be obtained until the app is deployed (needs the live URL). |
| **`requiredIf`** | string | v0.2 (finding 2) | Conditional requirement, e.g. `ENABLE_SECURITY_GATE=true` or a prose condition. |
| **`secretClass`** | `provider-paste\|create-then-copy\|internal-generated\|external-match` | v0.2 (finding 5 + review #6) | How the secret is sourced and synced (below). |

## Validation engine semantics (lives in the shared core, not the CLI)

- **`required` + `requiredIf`** — a key is required when `required: true` OR its `requiredIf`
  condition evaluates true. The engine must evaluate `requiredIf` against the other collected
  values / feature flags. (Finding 2: `PLATFORM_TRUST_*` only matters if `ENABLE_SECURITY_GATE=true`.)
- **`exclusiveGroup`** — exactly one member of a group must be satisfied; the others become
  not-required. **This redefines "required" for any consumer of the schema** — the deferred
  `feature-preflight` rewire must implement this semantically, not just parse it
  (this is the regression risk that moved the rewire to a follow-up, D4).
- **`keyFormat` is a shape hint, not validity.** Finding 4: Supabase `anon` and `service_role`
  are both `eyJ...` JWTs — the regex cannot tell a swapped paste apart. Therefore:
  - For JWT-class keys, additionally check the decoded `role` claim.
  - Offer optional **live-ping** validation (a cheap authenticated call per provider) — the only
    check that proves the key is valid, unrevoked, and has quota (review #8).

## Secret classes (review #6 + finding 5)

| `secretClass` | Source | Sync rule | Example |
|---|---|---|---|
| `provider-paste` | Copied from the provider dashboard | None | `ANTHROPIC_API_KEY` |
| `create-then-copy` | Generated by the provider AFTER you create a resource that needs the deployed URL | `phase: post-deploy` | `STRIPE_WEBHOOK_SECRET` |
| `internal-generated` | The wizard generates it; only this app consumes it | Safe to auto-gen | `CRON_SECRET` (none in mmc-application) |
| `external-match` | The wizard generates it AND an external sender must be configured with the SAME value | Wizard MUST instruct the operator to paste it into the sender, or it 401s silently | inbound-webhook shared secrets |

The `external-match` distinction is the fix for the silent-401 footgun: auto-generating a
secret and pushing it only to the app desyncs it from the sender.

## Wizard flow (consequence of `phase` + `provider`)

1. **Group by `provider`** — render one section per account, so the operator sets up Supabase /
   Stripe / etc. once and grabs all its keys together (finding 3).
2. **Two passes by `phase`:**
   - **Pass 1 (build + runtime):** collect everything obtainable before deploy.
   - **Pass 2 (post-deploy):** after the app has a URL, collect `dependsOnDeploy` keys
     (Stripe webhook secret) — the wizard prints the exact endpoint URL to register first.
3. **Per-key idempotency:** keep-existing by default; show the current value masked; `--force` to overwrite.

## Hard rules

- **Never log or echo a `secret`/`build-time` secret.** Pipe to adapters via stdin, never argv
  (the existing `vercel-env-restore.mjs` already does this — reuse it).
- **BYOK manifests must not hardcode CAS infra** (finding 6). Operator-owned values
  (`RESEND_FROM_EMAIL`, domains, project refs) stay generic in the manifest — no
  `updates.corporateaisolutions.com`, no CAS Supabase refs.
- **`.env.local` must be gitignored** — the env-file adapter refuses to write if it isn't.

## Distribution adapters

| Adapter | Status | Reuses |
|---|---|---|
| `env-file` | core (now) | writes `.env.local` (gitignore-checked, atomic) |
| `vercel` | core (now) | wraps `vercel-env-restore.mjs` (stdin push, per-target) — must be injectable for tests (review #10) |
| `supabase` | deferred | `configure-supabase-auth` route pattern (redirect URLs / SMTP) |

## Open schema questions (next)

- Does `requiredIf` need a real expression grammar, or is `KEY=value` + a prose fallback enough?
- Should `provider` carry its own setup URL + cost note (dedupe across that provider's keys)?
- Live-ping validators: per-provider plugins in the core, or a `validateUrl` per service entry?
